# 의사결정 기록

미래의 사람과 에이전트가 같은 결정을 다시 추론하지 않도록, 오래 남겨야 하는 결정을 기록합니다.

## 결정 템플릿

```markdown
## YYYY-MM-DD: 결정 제목

상태: 제안됨 | 채택됨 | 대체됨

### 배경

왜 이 결정을 해야 했는가?

### 결정

무엇을 선택했는가?

### 영향

무엇이 쉬워지고, 어려워지고, 제한되는가?

### 링크

- 관련 PRD 섹션:
- 관련 이슈/태스크:
```

## 2026-06-28: 제품 원본 문서는 스킬 밖에 둔다

상태: 채택됨

### 배경

프로젝트에는 재사용 가능한 에이전트 작업 규칙과, PRD/고객 모델/하네스 전략 같은 제품 원본 문서가 모두 필요합니다.

### 결정

재사용 가능한 에이전트 작업 방식은 `.codex/skills/our-medical-note-rules/`에 둡니다. 제품 원본 문서는 `docs/`에 둡니다.

### 영향

에이전트는 `AGENTS.md`에서 시작해 `docs/`와 관련 스킬 규칙을 읽습니다. 제품 맥락을 스킬 내부 참조에 숨기지 않습니다.

## 2026-06-28: 초기 제품 방향과 기술 방향

상태: 채택됨

### 배경

서비스의 실제 방향이 구체화되었습니다. 제품은 병원 방문 전후 기록, 그룹 공유, OCR 약 정보 기록, 장기 리포트를 중심으로 합니다.

### 결정

- 서비스명은 우리의 의료 노트(Our's Medical Note)입니다.
- Kakao 로그인만 고려합니다.
- 프론트엔드는 SEO를 위해 Next.js를 우선 검토합니다.
- 별도 백엔드 서버 없이 Supabase를 사용합니다.
- pnpm 모노레포로 진행합니다.
- `apps/web`, `apps/app`, `apps/admin`을 둡니다.
- `services/supabase`에 Supabase 관련 코드를 둡니다.
- `packages/ui`, `packages/storybook`, `packages/tailwind-config`, `packages/eslint-config`를 우선 둡니다.

### 영향

데이터 모델과 권한 모델은 그룹 공유와 기록 대상자 개념을 중심으로 설계해야 합니다. Supabase RLS 정책과 하네스 테스트에서 권한 누출을 중요한 회귀 케이스로 다루어야 합니다.
