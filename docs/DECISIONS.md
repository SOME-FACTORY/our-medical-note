## 2026-06-28: 제품 원본 문서는 스킬 밖에 둔다

상태: 채택됨

### 배경

프로젝트에는 재사용 가능한 에이전트 작업 규칙과, PRD/고객 모델/하네스 전략 같은 제품 원본 문서가 모두 필요합니다.

### 결정

재사용 가능한 에이전트 작업 방식은 `.codex/skills/our-medical-note-rules/`에 둡니다. 제품 원본 문서는 `docs/`에 둡니다.

### 영향

에이전트는 `AGENTS.md`에서 시작해 `docs/`와 관련 스킬 규칙을 읽습니다. 제품 맥락을 스킬 내부 참조에 숨기지 않습니다.

## 2026-06-28: 초기 제품 방향과 기술 방향

상태: 채택됨

### 배경

서비스의 실제 방향이 구체화되었습니다. 제품은 병원 방문 전후 기록, 그룹 공유, OCR 약 정보 기록, 장기 리포트를 중심으로 합니다.

### 결정

- 서비스명은 우리의 의료 노트(Our's Medical Note)입니다.
- Kakao 로그인만 고려합니다.
- 프론트엔드는 SEO를 위해 Next.js를 우선 검토합니다.
- 별도 백엔드 서버 없이 Supabase를 사용합니다.
- pnpm 모노레포로 진행합니다.
- `apps/web`, `apps/app`, `apps/admin`을 둡니다.
- `services/supabase`에 Supabase 관련 코드를 둡니다.
- `packages/ui`, `packages/storybook`, `packages/tailwind-config`, `packages/eslint-config`를 우선 둡니다.

### 영향

데이터 모델과 권한 모델은 그룹 공유와 기록 대상자 개념을 중심으로 설계해야 합니다. Supabase RLS 정책과 하네스 테스트에서 권한 누출을 중요한 회귀 케이스로 다루어야 합니다.

## 2026-06-29: 초기 Supabase 스키마는 사람과 로그인 계정을 분리한다

상태: 채택됨

### 배경

서비스는 실명 기반 가입 사용자를 다루지만, 의료 기록의 대상자는 앱에 가입한 사용자와 항상 같지 않습니다. 또한 그룹 안에서 각 구성원이 서로를 부르는 관계명은 그룹과 관찰자에 따라 달라질 수 있습니다.

### 결정

초기 Supabase 스키마는 `profiles`를 로그인 계정, `people`을 실제 사람과 기록 대상자, `care_group_members`를 그룹에 포함된 사람으로 분리합니다. 초대 링크와 QR은 `care_group_invites`와 `care_group_invite_acceptances`로 구성원 목록과 분리합니다. 그룹 내 관계 명칭은 `care_group_member_relations`에 `viewer_person_id`, `target_person_id`, `relation_name`으로 저장합니다. 의료 노트는 `medical_notes`와 `medical_note_group_shares`로 분리하고, `note_attachments`는 1차 스키마에 포함하지 않습니다.

### 영향

가입하지 않은 사람은 구성원 목록에 미리 노출하지 않고 초대 링크 상태로만 관리합니다. 초대를 수락한 가입 사용자만 그룹 구성원이 됩니다. 같은 대상자라도 그룹과 사용자 맥락에 따라 `엄마`, `딸`, `보호자` 같은 명칭을 다르게 저장할 수 있습니다. 의료 노트 공유는 선택된 그룹만을 통해 이루어지며, RLS와 공유 검증 트리거가 기록 대상자가 포함되지 않은 그룹 공유를 막습니다.

### 링크

- 관련 PRD 섹션: 초기 데이터 요구사항, 그룹 공유
- 관련 하네스: `harness/evals/group-permissions`
